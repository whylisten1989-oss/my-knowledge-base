### 为什么这样实现

当前网站是 Vue 3 CDN 静态多页面架构，没有 npm 构建流程。Customer BI 因此继续采用独立 HTML、CSS 和 JavaScript 文件，避免影响现有 Finance、Habits、Wiki、Profile 页面，也避免把项目改造成 Vite、Next.js 等全新工程。

Excel 先在浏览器本地解析，用户确认人员和结果后才写入数据库。这样可以防止文件里的无关人员被自动计入团队指标和排名，也可以在正式保存前发现字段问题。

### 本阶段完成内容

- 新增“昨日”范围解析：以数据库最新有效业务日期为基准，并找到前一个有效业务日用于对比。
- 新增“近 7 个有效业务日”范围解析：只取真实存在数据的日期，自动跳过空白自然日；对比范围取此前连续 7 个有效业务日。
- 新增“本月”范围解析：以最新有效业务日期所在自然月为当前月，并按相同日号截取上月同期。
- 新增按人员跨日聚合：满意率汇总好评和差评后计算，转化率汇总下单人数和询单人数后计算，工作时间平均响应时长按有效参与人日简单平均。
- 新增跨周期综合得分重新计算和排名，不读取或平均每日名次。
- 新增近 7 日至少参与 2 天、本月至少参与 50% 且不少于 3 天的资格标记。
- 新增本月只有 1 至 2 个业务日时的临时排行标记，不生成正式冠军结果。

### 为什么这样实现

时间范围和聚合被放在 `customer-bi-core.js` 的纯计算函数中，不依赖 Vue 页面或 Supabase。这样同一套口径可以同时供 Dashboard、排行榜和人员详情使用，也能直接用 Node.js 做断言测试，降低三个页面口径不一致的风险。

本月基准使用“最新 confirmed 快照所在月份”，而不是直接用电脑当前日期。这样当数据库暂时停止更新或当前月还没有任何数据时，页面不会凭空显示一个无数据月份；同时标题会展示真实截止日期。
