# 客服绩效 BI 阶段一设计方案

## 1. 修订后的业务理解

本系统服务于客服团队每日 KPI 绩效分析。数据源为固定格式 Excel，用户必须经过“上传、日期确认、人员选择、校验、计算预览、确认入库”后形成不可变的每日快照。

核心 KPI 暂定：

- 满意率；
- 平均响应时长；
- 转化率；
- 后续由管理员通过版本化规则新增的其他指标。

微信小店均响彻底排除：不计算、不评分、不排名、不展示、不生成趋势、不进入核心数据库字段。若 Excel 出现相关列，只记录为被忽略字段，其他列继续正常解析。

## 2. 四类人员范围

1. **原始文件人员**：Excel 中解析出的全部真实人员行。
2. **当前导入人员**：本次由用户勾选、确认参与统计的人。
3. **默认统计人员**：常用统计名单中的人员，只用于下一次自动预选。
4. **历史批次人员**：某一确认批次实际纳入或排除的人员，是历史排名和荣誉的固定依据。

未选择人员：

- 不写入 `bi_daily_metrics`；
- 不参与得分、排名、团队均值、周/月聚合和荣誉；
- 不视为未达标；
- 不影响连续达标或连续第一；
- 在 `bi_import_batch_agents` 中保留排除事实，用于审计和恢复选择。

## 3. 人员选择与常用名单

### 导入时的选择能力

- 全选、取消全选、反选；
- 搜索客服账号或昵称；
- 按团队、平台、名单筛选；
- 显示已选和已排除人数；
- 显示好评、差评、满意率、均响、询单、下单、转化率预览；
- 标记新人员、缺失人员、停用人员、未匹配人员；
- 使用默认名单、恢复最近一次选择、另存为新名单。

### 常用名单模型

- `bi_agent_groups` 保存“全部客服、售后团队、商丘团队、某平台团队”等名单。
- `bi_agent_group_members` 保存人员与名单的有效期关系。
- 名单可以设为默认；同一工作空间允许多个名单，但只有一个默认名单。
- 临时增减人员只改变本次批次，不自动改默认名单。
- 用户主动选择“保存为默认名单”或“加入名单”后才更新名单。
- 人员停用只更新 `bi_agents.is_active`，历史指标、排名和荣誉永久保留。
- 人员改名通过 `bi_agent_aliases` 继续映射到同一个 `agent_id`。

### 新人员与缺失人员

- Excel 首次出现且无法匹配的人员默认不勾选，提示“新人员”；用户可创建/匹配人员，并选择是否加入默认名单。
- 默认名单中存在但 Excel 缺失的人员提示“本次缺失”，不写入空成绩，不视为未达标。
- 历史存在但已停用的人员即使出现在 Excel 中也默认不勾选。

## 4. 五步导入向导

### 步骤 1：上传文件

- 拖拽或点击选择 `.xlsx`；
- 显示文件名、大小和 SHA-256；
- 识别模板版本；
- 完全相同文件哈希已确认时阻止重复导入。

### 步骤 2：选择业务日期

- 尝试从文件内容或文件名读取日期；
- 本样本没有日期列，因此必须由用户确认；
- 已存在有效批次时提示“创建修正版”，不能直接覆盖。

### 步骤 3：选择人员

- 解析全部人员后再选择；
- 使用默认名单进行预选；
- 保留每个人的选择来源：默认名单、最近选择、手动加入、手动排除、停用排除。

### 步骤 4：校验与计算预览

- 字段、类型、范围、身份映射、重复和历史冲突校验；
- 展示参与/排除人数、团队指标、预计前三名和预计荣誉；
- 有 error 时禁用确认；warning 必须可展开查看。

### 步骤 5：确认入库

- 展示业务日期、团队范围、人员清单、规则版本、批次 revision 和被替代批次；
- 二次确认后通过单个数据库事务写入；
- 写入失败则整批回滚。

## 6. KPI 计算

### 满意率

```text
满意率 = 有效好评数 ÷ (有效好评数 + 有效差评数)
```

分母为 0 时记为 `NULL/无评价`，不自动当作 0%。

距离目标还需好评数：

```text
ceil((目标满意率 × 当前有效评价数 - 当前好评数) ÷ (1 - 目标满意率))
```

结果至少为 0。目标为 100% 且已有差评时显示“有限新增好评无法达到”。

### 平均响应时长

- 已确认来源：`工作时间平响时长`，解析为秒；
- 值为 `-`、空或非法格式时记为无有效值；
- `0秒` 需要校验：无接待会话时不能自动认定为优秀成绩；
- 团队口径：只对被选中且数值有效的人取简单平均。

页面固定提示：

> 平均响应时长为当前所选人员的简单平均，暂未按会话量加权。

### 转化率

```text
转化率 = 下单人数 ÷ 询单人数
```

询单人数为 0 时记为 `NULL`。Excel 的 `询单转化率` 仅用于一致性校验。

### 综合得分

V1 规则版本为 `v1-2026-07-22`：满意率目标 90%、权重 50%；转化率目标 30%、权重 25%；工作时间均响目标 15 秒、权重 25%。各档评分为：

- 满意率：`≥92%` 得 110，`90%–92%` 得 100，`88%–90%` 得 80，`86%–88%` 得 60，低于 86% 得 0；
- 转化率：`≥30%` 得 110，`25%–30%` 得 100，`23%–25%` 得 80，`20%–23%` 得 60，低于 20% 得 30；
- 工作时间均响：`≤15秒` 得 110，`15–18秒` 得 100，`18–21秒` 得 80，高于 21 秒得 0。

```text
综合得分 = 满意率得分 × 权重
         + 响应时长得分 × 权重
         + 转化率得分 × 权重
         + 管理员新增指标得分 × 对应权重
```

三项任一核心指标无法计算时，V1 阻止该人员随本批次确认入库，避免不完整数据进入正式排名。

## 7. 团队指标与动态范围

团队满意率：

```text
Σ参与人员好评数 ÷ (Σ参与人员好评数 + Σ参与人员差评数)
```

团队转化率：

```text
Σ参与人员下单人数 ÷ Σ参与人员询单人数
```

团队平均响应时长：

```text
参与且均响有效人员的 avg_response_seconds 简单平均
```

人员范围切换规则：

- 未确认预览：选择变化后立即重算所有 Dashboard、预估排名、团队均值和荣誉。
- 已确认历史：默认进入“官方快照模式”，完全复现当时参与人员、排名和荣誉。
- 用户对历史日期切换人员名单时，进入“动态分析模式”，对“历史参与人员 ∩ 当前筛选人员”临时重算；必须显示“非官方重算”，不得覆盖正式排名和荣誉。

## 8. 排名方案

官方排名只使用 `bi_import_batch_agents.is_included = true` 且已产生有效 KPI 的人员。

默认排序：

1. 综合得分降序；
2. 满意率降序；
3. 平均响应时长升序；
4. 转化率降序；
5. 稳定 `agent_id` 用于确定性排序。

业务名次默认使用 `DENSE_RANK`。榜单标题必须包含日期、范围和人数，例如：

> 2026-07-22 商丘客服组，共 18 人参与排名

被排除或缺失人员没有当日排名，也不计为末位或未达标。

## 9. 日、周、月聚合

### 日

- 使用确认批次保存的参与人员范围；
- 保存官方团队汇总、排名和荣誉；
- 修正版产生新 revision，旧版本保留。

### 周/月团队指标

- 满意率：期间所有有效参与人日的好评数与差评数加总后计算；
- 转化率：期间所有有效参与人日的下单人数与询单人数加总后计算；
- 平均响应时长：期间所有有效参与人日的均响秒数简单平均，不对每日团队平均再次平均；
- 参与人数：使用去重客服数，同时展示“参与人日数”；
- 仅统计每个日期的当前有效 confirmed revision。

### 周/月个人指标

- 满意率和转化率按原始计数汇总后重新计算；
- 均响按该人员有效参与日简单平均；
- 综合分默认使用有效参与日每日得分的平均值；
- 周/月排名应设置最低参与天数，避免只参与一天的人直接夺冠。

若周期内 KPI 规则发生版本切换，报表应分段提示。正式周期排名原则上只比较同一规则版本；跨版本合并需要管理员明确允许。

## 10. 连续记录默认规则

合理默认：**未参与统计时暂停，不中断。**

- 参与且达标：连续达标次数 +1；
- 参与但未达标：连续达标清零；
- 休息、未排班、Excel 缺失或本次未纳入：暂停，既不增加也不清零；
- 连续第一同理：参与且第一增加，参与但非第一清零，未参与暂停。

规则需在 `bi_kpi_rules.streak_policy` 中可配置：

- `pause_on_absence`（默认）；
- `break_on_absence`；
- `break_after_n_calendar_days`。

## 11. 历史快照、去重与修正

### 去重

- 文件级：`workspace_id + file_hash` 防止完全相同文件重复确认；
- 批次内：`batch_id + source_row_key` 唯一；
- 人员级：`batch_id + agent_id` 唯一；
- 正式指标：`batch_id + agent_id` 唯一。

### 修正

- 已确认批次不可更新正式 KPI 行；
- 同日期同范围再次导入时创建 `snapshot_revision + 1`；
- 新批次记录 `supersedes_batch_id`；
- 原批次改为 `superseded`，但原指标、排名和荣誉不删除；
- 默认查询只读取当前有效 revision。

## 12. 荣誉系统

第一版建议：

- 综合冠军、团队冠军；
- 满意服务之星；
- 极速响应之星；
- 转化冠军；
- 全指标达标；
- 进步之星；
- 连续达标；
- 周冠军、月冠军。

荣誉只对当前确认批次的参与人员生成。荣誉规则必须包含最低样本量；例如满意率 100% 但只有 1 个评价时，不应默认获得满意服务之星。

荣誉记录绑定 `batch_id + rule_id + rule_version`。修正版生成新荣誉，旧荣誉仍归属旧批次。

## 13. 适配当前静态 Vue 3 CDN 架构

现有网站页面结构和生产业务代码保持不变。阶段二建议新增隔离页面：

```text
customer-bi.html
css/customer-bi.css
js/customer-bi/app.js
js/customer-bi/import.js
js/customer-bi/validation.js
js/customer-bi/calculations.js
js/customer-bi/api.js
js/customer-bi/charts.js
```

第一阶段只创建 `docs/customer-bi-design/`。

部署链路保持现状：在 `feature/customer-service-bi` 开发与评审，推送 GitHub 后先由 Vercel 生成 Preview；只有页面方案、导入结果和权限策略验收通过后才合并 `main`，由现有 Vercel Production 部署继续发布。Supabase URL 与 publishable key 应配置在 Vercel 环境变量中，不提交到 GitHub；Preview 和 Production 建议分别连接测试库与正式库，避免预览页面写入正式快照。

前端可使用 SheetJS 解析 Excel，并在浏览器完成即时人员选择和预览；正式确认、快照、排名和荣誉应由 Supabase RPC 在单个事务中执行，避免客户端篡改或部分写入。

Customer BI 使用独立 Supabase 项目，与现有 MindForge 数据库隔离。publishable key 允许在前端使用，但本阶段不写入仓库；任何 `service_role` key 都不得进入浏览器或 GitHub。

## 14. Supabase 停用/不可恢复迁移

- schema、migration、RLS、函数和默认规则全部进入 Git；
- 每个确认批次导出 `original.xlsx + normalized.csv + metrics.csv + rankings.csv + achievements.csv + manifest.json`；
- 原始文件和归档包同步到 Supabase 之外的加密对象存储；
- 定期执行 PostgreSQL 逻辑备份和恢复演练；
- 迁移目标可为新 Supabase、标准 PostgreSQL、Neon、Railway 或 RDS；
- Auth 密码不保证可移植，灾难恢复时可能要求用户重置密码；绩效和审计数据不应受影响；
- 前端仅需切换独立 BI 配置中的项目 URL 与 publishable key。
